# PR parity report

Post a translation parity table as a GitHub PR comment on every pull request.
Reviewers see exactly which strings were added, changed, or removed - without
needing to diff JSON files by hand. The job fails when any target locale is
missing a translation, has orphaned keys, or contains invalid ICU.

## How it works

`autotranslate parity` compares the committed catalog at the PR base ref against
the working-tree catalog in the PR branch. It outputs a markdown table (one
column per target locale) and exits non-zero when parity problems are found. The
GitHub Actions workflow below posts that output as a comment and fails the check
when the exit code is non-zero.

## Workflow

```yaml
# .github/workflows/parity.yml
name: Translation parity

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  contents: read
  pull-requests: write

jobs:
  parity:
    name: Catalog parity
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4

      - uses: actions/setup-node@v6
        with:
          node-version-file: .nvmrc
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - name: Run parity check
        id: parity
        run: |
          npx autotranslate parity \
            --base origin/${{ github.base_ref }} \
            --format github \
            > parity-report.md; echo "exit=$?" >> "$GITHUB_OUTPUT"
        continue-on-error: true

      - name: Post PR comment
        env:
          GH_TOKEN: ${{ github.token }}
          PR: ${{ github.event.pull_request.number }}
        run: |
          gh pr comment "$PR" \
            --body-file parity-report.md \
            --edit-last \
          || gh pr comment "$PR" \
            --body-file parity-report.md

      - name: Fail on parity problems
        if: steps.parity.outputs.exit != '0'
        run: exit 1
```

The `fetch-depth: 0` checkout gives git access to
`origin/${{ github.base_ref }}` so `parity` can read the base catalog via
`git show`.

## What the comment looks like

```markdown
## autotranslate - catalog parity

**2 strings added / 1 changed - all 3 locales complete**

| source (en)               | es             | fr              | ja               |
| ------------------------- | -------------- | --------------- | ---------------- |
| Check out                 | Pagar          | Payer           | チェックアウト   |
| ~~Empty cart~~ Clear cart | Vaciar carrito | Vider le panier | カートを空にする |

<sub>generated by autotranslate parity</sub>
```

## Options

| Flag                      | Default       | Description                                  |
| ------------------------- | ------------- | -------------------------------------------- |
| `--base <ref>`            | `origin/main` | Git ref to compare against.                  |
| `--format <text\|github>` | `text`        | Output format. Use `github` for PR comments. |

## Exit codes

| Code | Meaning                                                                       |
| ---- | ----------------------------------------------------------------------------- |
| `0`  | Catalog is complete - all target locales in sync.                             |
| `1`  | One or more parity problems: missing translation, orphan key, or invalid ICU. |

Added strings that are fully translated never trigger a non-zero exit. Only
genuine catalog problems (missing, orphan, invalid-icu) cause the job to fail.

## Tips

- **Draft PRs:** Add `if: github.event.pull_request.draft == false` to skip the
  job while drafts are in progress and the catalog is still being filled.
- **Required status check:** Go to branch protection settings and add
  "Translation parity" as a required check to block merging with broken
  catalogs.
- **Large catalogs:** The table is capped at 50 rows. A `+N more` line appears
  at the bottom when the diff exceeds the cap.
- **No API key needed in CI:** `parity` only reads existing catalog files from
  git - it never calls a translation provider.
