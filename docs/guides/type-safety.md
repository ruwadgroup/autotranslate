# Type safety

`autotranslate generate-types` codegens a `.d.ts` that augments
`@autotranslate/react` with your literal catalog keys. After it runs,
`useT('Sing out')` becomes a TypeScript error.

## How it works

`@autotranslate/react` exports an open `AutotranslateCatalog` interface:

```ts
// biome-ignore lint/suspicious/noEmptyInterface: open for module augmentation
export interface AutotranslateCatalog {}

export type CatalogKey = keyof AutotranslateCatalog extends never
  ? string
  : keyof AutotranslateCatalog | (string & {});
```

When `keyof AutotranslateCatalog` is `never` (no augmentation yet), `CatalogKey`
falls back to plain `string` so existing code compiles. After augmentation, it
narrows to your key set with `(string & {})` to preserve autocomplete without
rejecting unknown strings outright.

## Generate

```bash
npx autotranslate generate-types
```

Output: `<outDir>/types.d.ts`. Example contents:

```ts
// Generated by `autotranslate generate-types`. Do not edit.
import '@autotranslate/react';

declare module '@autotranslate/react' {
  interface AutotranslateCatalog {
    'Sign out': true;
    'Hello, {name}!': true;
    't.abc123def456': true;
  }
}
```

## Wire it up

Reference the file from your `tsconfig.json`:

```jsonc
{
  "compilerOptions": { "strict": true },
  "include": ["src", ".translations/types.d.ts"],
}
```

Or just include the entire `.translations` directory:

```jsonc
{
  "include": ["src", ".translations"],
}
```

## Run on every extract

Add the typegen to your `i18n` script so the type set stays in sync:

```jsonc
{
  "scripts": {
    "i18n": "autotranslate extract && autotranslate translate && autotranslate generate-types",
  },
}
```

CI runs `pnpm i18n` (or just `pnpm typecheck` after the catalogs land) and fails
if any new code references a key the catalog doesn't have.

## Catch errors

```ts
const t = useT();

t('Sign out'); // ✅
t('Sing out'); // ❌ Argument of type '"Sing out"' is not assignable …
t('Welcome, {name}!'); // ✅
```

Unknown strings still type-check (that's the `(string & {})` arm), but your
editor's autocomplete surfaces only the catalog keys.

## Limits

- **Dynamic keys aren't tracked.** If you use `t(key)` where `key` is a
  variable, the type system has no way to verify it. The
  [`no-dynamic-key`](eslint.md) ESLint rule rejects this at lint time.

- **ICU param inference is partial.** `useT` accepts
  `Readonly<Record<string, unknown>>` for `params`. Tighter param-shape
  inference per key is on the v1 roadmap.

- **`<T>` keys are hashes.** `t.abc123` keys appear in the augmentation but you
  don't reference them directly in code — they're how the runtime looks the tree
  up. Their presence in the union doesn't matter for typed call sites.

## Tips

- **Re-run typegen on every catalog change.** Wire it into your pre-commit hook
  or `i18n` script so stale types don't slip through.

- **Don't check `types.d.ts` into git.** Add `.translations/types.d.ts` (or the
  entire `.translations/.cache/` and friends) to `.gitignore` if you regenerate
  it in CI; commit it if you want zero-config local builds. Either choice is
  valid; pick one and stick with it.

- **Combine with the ESLint plugin.** `valid-icu-format` parses each key as ICU
  and rejects malformed messages before runtime, complementing the type-level
  check.
